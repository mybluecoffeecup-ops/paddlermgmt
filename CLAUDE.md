## Commands

```bash
npm run dev      # start dev server (Turbopack) at http://localhost:3000
npm run build    # production build — also runs the TypeScript check
npm run start    # run a production build
npm run lint     # ESLint (flat config: eslint-config-next core-web-vitals + typescript)
```

There is no test suite configured in this repo.

Global npm config on this machine pins npm's `allow-scripts` security feature and disallows project-level overrides — plain `npm install` works (scripts run only for allow-listed packages), but anything that passes `--allow-scripts` explicitly (e.g. some `create-*` scaffolding tools) will fail with `EALLOWSCRIPTS`.

## Architecture

Next.js App Router + TypeScript + Tailwind v4, mobile/iPad-first, backed by Supabase/Postgres.

### Data layer: one context, every `(app)` route

`src/hooks/app-data.tsx` (`AppDataProvider`/`useAppData`) is the single source of truth for all domain data (profiles, sessions, attendance, races, race_commitments, calendar_events, lineups) and is mounted in `src/app/(app)/layout.tsx`, wrapping every route in that route group (`/`, `/sessions`, `/calendar`, `/lineups/*`, `/profile`, `/info`, `/orders`). The root `src/app/layout.tsx` only sets up fonts and the `<html>`/`<body>` shell — it does *not* mount the provider, so `/login` and `/signup` (outside the `(app)` group) have no access to `useAppData()` by design.

- State is seeded from `src/lib/mock-data.ts` and used as-is when Supabase isn't configured.
- `src/lib/supabase/client.ts` exports `isSupabaseConfigured` (true when `NEXT_PUBLIC_SUPABASE_URL` + `NEXT_PUBLIC_SUPABASE_ANON_KEY` are set). When true, `AppDataProvider` fetches live rows on mount and every mutator (`rsvpToSession`, `updateProfile`, `updateSession`, `createSession`, `updateRaceCommitment`, `createLineup`, `saveLineupSeating`) updates local state optimistically *and* fires the matching `src/lib/api/*.ts` CRUD call. API failures are logged, not rolled back.
- `src/lib/supabase/database.types.ts` documents the table shapes but is **not** passed as a generic to `createBrowserClient` — supabase-js's `GenericTable` constraints didn't structurally match hand-written row types without heavy boilerplate, so each `api/*.ts` function casts its result to the domain type from `src/types/index.ts` instead.
- Domain types in `src/types/index.ts` mirror the SQL schema 1:1; treat that file and the migrations as the source of truth together.
- **Async resolution race**: `role` is derived from `currentUser?.is_coach`, and `currentUser` resolves asynchronously (a profiles fetch plus a separate Supabase auth effect). Before both settle, `currentUser` is `undefined` and `role` silently falls back to `"paddler"` — even for an actual coach. Anything that gates behavior on `role` must wait for `!loading && currentUser !== undefined` before trusting it, or a coach will get bounced/flash-hidden on first load. See `src/components/auth/RequireCoach.tsx` and the nav filtering in `AppShell.tsx` for the established pattern.

### Auth and roles

Real Supabase Auth (email/password) is wired up: `/login` and `/signup` (`src/app/login/page.tsx`, `src/app/signup/page.tsx`) call server actions in `src/lib/auth-actions.ts`; `/auth/callback` exchanges the email-confirmation code for a session; new signups get a `profiles` row auto-created by the `handle_new_user()` trigger (`supabase/migrations/0002_handle_new_user.sql`) with `is_coach` defaulting to `false`.

Route protection is in `src/proxy.ts` — **not** `middleware.ts`. This is a real Next.js 16 rename (Middleware → Proxy, same mechanism, `export function proxy` instead of `export function middleware`); don't "fix" the filename. It only checks whether a user is logged in and redirects to `/login`/`/`; it does **not** check role. `role: "paddler" | "coach"` is derived from `currentUser.is_coach` (see the race condition above) — a coach is also a full paddler in the data model (same `profiles` row, same attendance/weight/etc.), so `/`, `/profile`, `/calendar`, `/info`, and `/orders` are intentionally *not* role-gated.

Coach-only route gating is client-side via `RequireCoach` (`src/components/auth/RequireCoach.tsx`), but it's wired at two different levels depending on whether the segment has a paddler-visible child route:
- `/lineups` (and `/lineups/[lineupId]`, both fully coach-only) uses a **`layout.tsx`** at `src/app/(app)/lineups/layout.tsx` — the gate cascades to every route under that segment, which is fine here since there's nothing under `/lineups` that paddlers should see.
- `/sessions` is fully coach-only too, but `src/app/(app)/sessions/[sessionId]/page.tsx` (the session detail view) is **not** — paddlers can view/comment on it. So `RequireCoach` is wrapped **inline inside `src/app/(app)/sessions/page.tsx`'s own return**, not in a `sessions/layout.tsx` — a layout there would incorrectly cascade the gate down to `/sessions/[sessionId]` too.
- `/calendar` is a hybrid: the page itself is open to both roles (everyone needs to view the calendar), but the coach-only `RaceManager`/`CalendarEventManager` add/edit panels within it are conditionally rendered with an inline `resolved && role === "coach"` check (mirroring `RequireCoach`'s internal logic, see the race condition note above) rather than being wrapped in `RequireCoach` itself — `RequireCoach` redirects non-coaches away entirely, which is wrong here since paddlers should still land on `/calendar`.

If Supabase isn't configured, `proxy.ts` is a no-op (no login required) and `AppShell` shows a manual Paddler/Coach toggle instead of a real session, flipping `currentUserId` between two fixed mock IDs (`CURRENT_USER_ID` / `CURRENT_COACH_ID` in `src/lib/mock-data.ts`) — this is the fast path for local UI work.

**Switching between mock and live mode locally**: since `isSupabaseConfigured` is read once at server start, there's no in-app toggle for it — comment out (or delete) `NEXT_PUBLIC_SUPABASE_URL`/`NEXT_PUBLIC_SUPABASE_ANON_KEY` in `.env.local` and restart `npm run dev` to force mock mode (the Paddler/Coach toggle reappears); uncomment and restart again to go back to live mode. `MOCK_PROFILES` in `src/lib/mock-data.ts` currently seeds 30 paddlers with full field variety (all `ALL_CREW_TAGS` values included) for exercising roster/filter/lineup UI without touching the real database.

### Design system: "Kit Green & Gold"

The visual identity is a flat, sporty, high-contrast look tuned for outdoor/sunlight readability — a green-and-gold palette inspired by national team kit colors, editorial and minimalist rather than soft SaaS defaults. **No drop shadows, no gradients, no backdrop-blur anywhere** — panel definition comes from a solid 2px border, not elevation.

Color tokens are defined inside an **`@theme { ... }` block** in `src/app/globals.css` and consumed through Tailwind v4's automatic utility generation (so plain `green-*`/`gold-*` classes everywhere already pick up the palette — no need to hunt down every usage). **They must stay inside `@theme`, not plain `:root`** — this broke once already: a `--color-green-700` declared in `:root` does *not* make `bg-green-700` use it, because `green` is already a stock Tailwind color family, so the class silently falls back to Tailwind's own built-in green (wrong hex, looks "close enough" to pass a casual glance) — and non-stock names (`gold`/`pitch`/`redcard`/`ink`/`chalk`) generate no utility at all in `:root`, rendering as invisible/transparent (white-on-white). If a future color token edit seems to have no visual effect, check it's still inside `@theme` first.
- **`green-*`** — primary brand/action color, a vivid Kit Green. `green-700` is the AAA-contrast (7.9:1 white text) "solid fill" shade used for buttons/active states/badges; `green-500`/`600` are for borders, icons, and text-on-white accents.
- **`gold-*`** — secondary accent and the "caution" semantic (payment due, imbalance warnings). **Rule: gold is always a solid fill with Ink text on top, never gold text on a light background** — the lighter/mid gold steps fail text contrast.
- **`pitch-*`** — deep green-black. Used for `AppShell`'s nav bar (a constant "team color" toolbar in both light and dark mode) and as the dark-mode background.
- **`redcard-*`** — functional-only alert color (Not Going, destructive, errors), never decorative.
- **`ink`** / **`chalk`** — single-value flat tokens (`#101410` / `#ffffff`) for primary text/panel borders and background, wired as `--background`/`--foreground` (so `bg-background`/`text-foreground` utilities exist too).

The **status badge system** in `src/components/ui/Badge.tsx` is the signature element: `AttendanceBadge`/`DisciplineBadge`/`CalendarCategoryBadge` all render soft tinted pills (translucent `bg-*-500/15`-to-`/25` washes with dark text, not solid fill) — strong solid color is reserved for meaningful, singular moments (an active RSVP selection), not passive repeated labels that appear many times per screen (roster rows, race lists, calendar agenda rows). Going/DB → green wash; Not Going → redcard wash; Payment due/OC → gold wash with dark text; Maybe → Ink-outlined with no fill. Since this one file controls status/category coloring everywhere (roster table, race RSVPs, session attendance, the calendar), keep new status UI going through it rather than hand-rolling colors.

Shape language: `rounded-full` is reserved for genuinely circular elements (the avatar, small side-indicator dots, circular RSVP buttons) — everything else uses a small, deliberate radius scale (`rounded-md` for panels, `rounded` for buttons/inputs/icon chips/tags). Barlow Condensed (loaded in `src/app/layout.tsx` as `--font-barlow-condensed`, exposed as `font-display`) is used for headings, panel titles, and numerals app-wide — keep leaning on it for anything condensed-caps/numeric. Body text uses **Barlow** (regular width, `--font-barlow`/`font-sans`), the same type family as the display face at a different optical width, loaded at medium/semibold/bold weights only (no light/regular 400 — thin weights are hard to read in bright glare).

Contrast is a hard constraint, not a nice-to-have: secondary/label text is never `slate-400`/`slate-500` on a light background (that's exactly what disappears in sunlight glare) — use `slate-600`/`slate-700` minimum. New UI should follow this: `font-display` uppercase-tracking-wide for headers/numbers, `focus-visible:ring-2 focus-visible:ring-green-500` on interactive elements, a solid `border-2 border-ink` (not a shadow) for panel definition, and green/gold/pitch/redcard (not arbitrary new hues, not stock Tailwind `emerald`/`amber`/`rose`) for anything semantically colored.

The app's brand name is **"PADDLES UP"** (tagline "Crew Mgmt") and its mark is a crossed dragon-boat paddle and outrigger paddle, `PaddlesIcon` (`src/components/ui/PaddlesIcon.tsx`) — a lucide-style single-`currentColor` SVG sized for the gold-500/ink header badge. `BrandMark` (`src/components/nav/BrandMark.tsx`) wraps the icon badge + name + tagline and is the single source for that block, reused by `AppShell`'s header (`variant="header"`, white text on the pitch nav bar) and by `/login`/`/signup` (`variant="card"`, dark text on a white Card) — edit `BrandMark`, not each call site, if the brand block itself changes. The standalone favicon/PWA icon is a self-contained variant of the same mark (own gold-500 background + ink paddles) at `src/app/icon.svg` (App Router favicon convention) and `public/icon.svg` (referenced by `public/manifest.json`'s PWA `icons` array) — keep these two files in sync if the mark changes. Do **not** confuse this brand name with the "Paddler" *role* label (the Paddler/Coach toggle buttons, the `profiles` roster column, etc.) — that's a domain-model role name, unrelated to the product brand, and stays as-is.

Placeholder/work-in-progress pages (`/info`, `/orders`) use the shared `WorkInProgress` component (`src/components/ui/WorkInProgress.tsx`) — a gold-500/ink badge with a `Construction` icon, following the gold-as-caution semantic above. Use it rather than hand-rolling another empty-state treatment for future placeholder pages. (The Calendar page's old "Race Signups" WIP placeholder was removed once the Calendar itself became real — races surfaced there link straight through to `/races/[raceId]` for RSVP.)

`prototype.html` is a **separate, static, single-file mockup** (its own inline Tailwind CDN config with its *own* teal/ember color tokens, its own hand-rolled JS state and hand-drawn SVG icons, no shared code with the Next.js app) that intentionally mirrors the real app's dashboard/command-center/lineup UI for demo purposes. It has no auth/role model. **It currently still has the old regatta-teal look, the old "Paddler Management" branding, and the pre-restructure three-tab layout — it has not been updated to Kit Green & Gold, the PADDLES UP rebrand, or the six-page nav** — that parity gap is deliberate (a separate follow-up pass), not an oversight; don't assume the two match right now. When making further visual or structural changes to the Next.js app, still check whether `prototype.html` needs the equivalent change to stay in parity going forward — it won't happen automatically. It's deployed to GitHub Pages via `.github/workflows/deploy-pages.yml` on every push to `main` (copies `prototype.html` to `dist/index.html`; the Next.js app itself isn't statically exported and isn't deployed there).

### Database (`supabase/migrations/`)

`0001_init.sql`: enums (`discipline_type`, `paddle_side`, `boat_type`, `attendance_status`) plus `profiles`, `sessions`, `attendance`, `races`, `race_commitments`, `lineups`, all with `updated_at` triggers and RLS (an `is_coach()` SQL helper gates writes to coach-only tables; paddlers can only write their own attendance/race_commitments rows). `0002_handle_new_user.sql`: the `handle_new_user()` trigger that creates a `profiles` row on signup. `0003_profile_eligibility_age_steer.sql`: replaces the old `is_pr_or_citizen` boolean with an `eligibility` enum (`Citizen`/`PR`/`Other`), splits the single `is_steer` boolean into boat-specific `is_oc_steer`/`is_db_steer`, and adds an optional `age_range` enum (`Under 40`/`40-50`/`50-60`/`60+`) — best-effort-migrates existing boolean values before dropping the old columns. `0004_session_type_enum.sql`: converts `sessions.type` from free text to a `session_type` enum (`Training`/`Race Simulation`/`Race`), best-effort-mapping old free-text values (e.g. `'Race Prep'` → `'Race Simulation'`, anything unrecognized → `'Training'`) before the column type change.

Note the RLS gap: `profiles_update_own_or_coach` allows a user to update *any column* on their own row — there's no column-level restriction. For `is_coach` specifically, `0019_restrict_is_coach_column.sql` closes this with a `before update` trigger (`enforce_is_coach_change()`) that rejects any change to `is_coach` unless the acting user is already a coach (via the existing `is_coach()` helper, which checks the *actor's* row, so it works whether a coach is editing their own row or another paddler's). That trigger is what makes the coach-only "promote/demote" control in `RosterTable.tsx` (`src/components/command-center/RosterTable.tsx`) safe to ship — it isn't relying on the UI alone. `benchmarks` and `coaching_feedback` have no equivalent DB-level protection yet: the app still compensates for those purely by never sending them in any self-service `updateProfile` patch (see `src/app/(app)/profile/page.tsx`); don't add a form field for either without also adding the same kind of guard.

`0010_calendar_events.sql`: adds the `calendar_events` table (see "Calendar" below) with the same `select-authenticated` / `write-coach` RLS shape as `races`. `0011_calendar_events_seed.sql`: seeds it with the club's real 2026 calendar (`created_by` left `null` — no real coach UUID exists at migration time).

Apply migrations via `supabase db push` or paste them into the SQL editor after linking a project, then copy `.env.local.example` to `.env.local`.

### Boat layouts and the lineup generator

`src/lib/boat-config.ts` defines `BOAT_LAYOUTS` (seat lists) for `DB12`/`DB22`/`V6` and `getBalanceGroups()`, which decides what the telemetry widget compares: Left vs Right seats for dragon boats, Odd vs Even seat number for the single-file `V6` (paddlers alternate paddling sides by seat position, and the steersperson's seat is included since their weight affects trim too). `WEIGHT_IMBALANCE_WARNING_KG` (15) is the threshold used by `TelemetryBar`.

`Lineup.seating_configuration` is a flat `seatId -> paddlerId | null` map (see `SeatingConfiguration` in `src/types/index.ts`). Bench membership is **derived, not stored**: it's "attending paddlers for this session" minus "paddler IDs currently present as a value in the seating map." The drag-and-drop logic lives entirely in `src/components/lineup/LineupEditor.tsx`'s `handleDragEnd` — dragging onto an occupied seat swaps the two occupants if the dragged paddler had a previous seat, or bumps the occupant back to the bench if the dragged paddler came from the bench. dnd-kit draggable IDs are paddler IDs; droppable IDs are seat IDs (plus the sentinel `BENCH_DROPPABLE_ID`).

### Session creation

`SessionManager.tsx` (rendered on the Session Mgmt page, `/sessions`; its component directory is still named `src/components/command-center/` for historical reasons — that directory is shared by both `/sessions` and `/calendar` (which renders `RaceManager` from there), it isn't a leftover) owns both browsing sessions and creating them: a "+ New Session" button in its `CardHeader` toggles an inline `NewSessionForm` that *replaces* the session-picker list while open (no modal component exists anywhere in this codebase — every "create" flow is an inline `Card` form, not a popup). `type` is a closed `SessionType` union (`"Training" | "Race Simulation" | "Race"`, `SESSION_TYPE_OPTIONS` in `src/types/index.ts`) backed by the `session_type` Postgres enum (`0004_session_type_enum.sql`) — unlike most other fields, this one used to be a free-text column and was tightened after the fact, so don't assume every string is a valid session type without checking this union first.

The "Repeat weekly" option is a thin client-side convenience, not a real recurrence feature: checking it just calls the `createSession` mutator (`src/hooks/app-data.tsx`) once per week in a loop (2–12 occurrences), each with `session_date` advanced by 7 days via `addDaysIso` (`src/lib/utils.ts`). `createSession` never throws — it updates local state and returns synchronously regardless of whether the background Supabase insert (when live) succeeds, so the loop can't detect partial failures and doesn't try to. Session IDs are `session-${Date.now()}-${crypto.randomUUID()}`; the timestamp alone isn't sufficient because synchronous loop iterations can share the same millisecond (this caused a real duplicate-React-key bug before the `randomUUID()` suffix was added — don't drop it if touching this code).

### Calendar

`calendar_events` (`0010_calendar_events.sql`) is deliberately **separate from `races`**, not a superset or replacement of it. `races` stays reserved for races the club is actually fielding a boat for — RSVP-capable via `race_commitments`, lineup-linkable — created deliberately through `RaceManager`'s form. `calendar_events` is lighter-weight, read-by-everyone, coach-editable reference data with no RSVP: training blocks, socials, public holidays, membership deadlines, club meetings, and any race-like text that isn't worth promoting into a full `Race` row. Don't collapse the two into one table — `races` has FKs from `race_commitments` and `lineups` that `calendar_events` was never meant to carry.

The `/calendar` page (`src/app/(app)/calendar/page.tsx`) renders both sources merged into one list via `buildCalendarItems(races, calendarEvents)` (`src/lib/calendar-utils.ts`), which maps `races` rows to `category: "Race"` items carrying an `href` to `/races/[raceId]` (so clicking one still lands on the real RSVP flow), and `calendar_events` rows to items with no `href` (informational only, or a jump-to-date action instead — see below).

**Layout is Outlook-style, not a stacked 12-month agenda**: a sidebar holds `CalendarMiniMonth` (one month's compact dot-grid, with prev/next chevrons) plus `CalendarUpcomingList` (next 10 items, `upcomingItems()`, scrollable); the main panel holds `CalendarMonthGrid` (a real 7-column day grid with up to 2 event chips per cell + "+N more", desktop-only via its own internal `hidden md:block` — there's no separate mobile variant, it just doesn't render below `md`) and `CalendarDayDetail` (the selected day's full item list, via `itemsForDay()`). All four components are **rendered exactly once** in `page.tsx`; getting sidebar-vs-main visual grouping right across the breakpoint where the layout goes from 1 column to 2 uses explicit `xl:col-start`/`xl:row-start` per child rather than an `order-*` chain, since a plain `order` chain can't express "these two are a column together" once the column count itself changes. On mobile (single column, grid hidden), plain DOM order already produces the right stack — MiniMonth → DayDetail → Upcoming — with no extra classes needed.

`CalendarUpcomingList` reuses the exact mobile-collapse pattern from `FilterSidebar.tsx` (`useState` open/closed + rotating `ChevronDown`, forced open via `xl:flex` regardless of state) rather than inventing a new one. `CalendarEventRow` (`src/components/calendar/CalendarEventRow.tsx`) is the shared title+badge row used by both `CalendarDayDetail` and `CalendarUpcomingList`: it renders as a `Link` when `item.href` is set (real races), or as a button calling `onSelect(dateIso)` otherwise — clicking a non-race upcoming item jumps the sidebar/grid to that date rather than doing nothing.

Selection state (`selectedYear`/`selectedMonth`/`selectedDate`) lives in `page.tsx`, initialized to *today* (`new Date()` / `todayIso()`), not hardcoded to 2026 — prev/next month rolls over Dec↔Jan into adjacent years correctly, since there's no reason to hard-lock navigation to whichever year happens to have seeded data. `buildMonthGridCells(year, month)` (leading blanks + day numbers) is shared by `CalendarMiniMonth` and `CalendarMonthGrid` so both grids agree on cell placement.

Category → color mapping lives in `src/components/ui/Badge.tsx` (`CALENDAR_CATEGORY_STYLES`/`CALENDAR_CATEGORY_DOT_STYLES`/`CalendarCategoryBadge`), following the same soft-tinted-pill convention as `AttendanceBadge`/`DisciplineBadge` in that file. Only `green`/`gold`/`pitch`/ink-outline are used (no new hues), so three of the seven categories intentionally share a color family and are differentiated by label text (in badges/rows) or icon-adjacent position (dots) rather than a unique hue each (Social/Deadline both gold; Meeting/Holiday/Other all ink-outlined/slate).

2026 is currently the only year with real seed data (`MOCK_CALENDAR_EVENTS` in `src/lib/mock-data.ts`, from the club's actual spreadsheet calendar via a one-time parse — not a runtime import feature) — browsing into other years works, they'll just show empty grids/mini-months until more years get seeded.

### Routes

All under `src/app/(app)/` (wrapped in `AppDataProvider` + `AppShell`, see above) unless noted. `AppShell`'s `NAV_ITEMS` array (`src/components/nav/AppShell.tsx`) is the source of truth for which of these appear in the top/bottom nav and their per-item `coachOnly` gating — add new top-level pages there.

- `/` — Home / paddler dashboard (`page.tsx`); role-agnostic, works identically for coaches (who are also paddlers)
- `/sessions` — Session Mgmt: session manager, metrics, segment roster filter/table, weekly workout program; coach-only, gated by an inline `<RequireCoach>` wrap inside `sessions/page.tsx` itself (see the gating note above — not a `layout.tsx`, since that would also gate the paddler-visible `/sessions/[sessionId]`)
- `/calendar` — Calendar: an Outlook-style single-month sidebar (`CalendarMiniMonth` + `CalendarUpcomingList`) alongside a full 7-column day grid (`CalendarMonthGrid`, desktop-only) and a selected-day detail panel (`CalendarDayDetail`), merging real races with the broader `calendar_events` table (see "Calendar" below) via `buildCalendarItems()` (`src/lib/calendar-utils.ts`). Open to both roles; the coach-only `RaceManager`/`CalendarEventManager` add/edit panels are conditionally rendered inline (see the gating note above)
- `/races/[raceId]` — race detail: RSVP (Going/Maybe/Not Going), payment status, linked lineups, comments; not role-gated, same shape as `/sessions/[sessionId]`
- `/lineups` — session/boat picker + list of existing lineups; gated by `lineups/layout.tsx` → `RequireCoach` (covers `/lineups/[lineupId]` too via layout nesting)
- `/lineups/[lineupId]` — the drag-and-drop editor, an async server component that awaits `params` and renders the client `LineupEditor`
- `/profile` — self-service profile form (name, weight, discipline, preferred side, eligibility, age range, boat roles [pacer/drummer/OC steer/DB steer], crew tags); not role-gated, reachable via the avatar link in `AppShell`'s header
- `/info` — work-in-progress placeholder (team information, reference documents); not role-gated
- `/orders` — work-in-progress placeholder (team gear/apparel ordering); not role-gated
- `/login`, `/signup` — outside the `(app)` group, no `AppDataProvider`/`AppShell`, use server actions from `src/lib/auth-actions.ts`
